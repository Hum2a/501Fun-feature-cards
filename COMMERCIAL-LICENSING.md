# Commercial licensing

The *feature-cards* codebase is published under **[AGPL-3.0-only](LICENSE)**.
That licence is ideal for open evaluation and copyleft network deployments, but
**not** for organisations that need:

- Closed-source integration
- No obligation to publish modified source to end users
- Embedded use in proprietary products without AGPL contagion
- White-label deployment without copyleft

## Dual-licensing model

| Use case | Licence path |
| --- | --- |
| Open source, copyleft OK, network source offer OK | **AGPL-3.0-only** — use freely under [LICENSE](LICENSE) |
| Proprietary / closed / no source offer | **Commercial licence** — contact author |
| Evaluation only | AGPL applies; no separate licence needed for reading and testing |

## How to request a commercial licence

Email **Humzab1711@hotmail.com** with subject `Commercial licence: feature-cards`.

Include:

1. Organisation name and site
2. Intended use (internal tool, SaaS, embedded in product, etc.)
3. Whether modifications are required
4. Approximate deployment scale (users, sites, developers)
5. Required term (perpetual vs subscription)

You will receive terms covering scope, attribution requirements (if any), fees,
and jurisdiction. Until a signed agreement exists, **only AGPL terms apply**.

## Why AGPL by default

See [ADR-0004](docs/adr/0004-agpl-licence.md). Portfolio and hiring evaluators
may use the project freely; unauthorised commercial closed-source reuse is
**legally actionable** as a licence violation rather than “ permitted by MIT ”.
